Tool Communication Protocol

Overview

This tool demonstrates the Epistemic Toolbox communication protocol. Tools receive commands from the viewer and emit events back, enabling bidirectional communication while maintaining CSS isolation via Shadow DOM.

Viewer-to-Tool Commands

Tools should listen for these custom events dispatched by the viewer:

tool-openHelp

  • Purpose: Opens the tool's help/documentation
  • Detail: No additional data required

tool-reset

  • Purpose: Resets the tool to its initial state
  • Detail: No additional data required
  • Response: Tool should emit tool-stateChanged with action "reset"

tool-refresh

  • Purpose: Refreshes or reloads tool data
  • Detail: No additional data required
  • Response: Tool should emit tool-stateChanged with action "refresh"

tool-[custom]

  • Purpose: Tool-specific custom commands
  • Detail: Custom data in event.detail object
  • Example: tool-exportData with format specification

Tool-to-Viewer Events

Tools should emit these events to communicate with the viewer:

tool-ready

  • Purpose: Signals that the tool is fully loaded and initialized
  • When to emit: After all scripts have executed and tool is ready
  • Detail object: { name: string, version: string, [metadata] }

tool-stateChanged

  • Purpose: Reports significant state changes to the viewer
  • When to emit: After reset, refresh, or major state updates
  • Detail object: { action: string, [stateData] }

tool-error

  • Purpose: Reports errors to the viewer
  • When to emit: When unrecoverable errors occur
  • Detail object: { message: string, [errorDetails] }

Implementation Example

// Listen for viewer commands
document.addEventListener('tool-reset', (e) => {
    // Reset tool state
    resetTool();
    
    // Notify viewer
    document.dispatchEvent(new CustomEvent('tool-stateChanged', {
        detail: { action: 'reset', timestamp: Date.now() },
        bubbles: true,
        composed: true
    }));
});

// Notify viewer when ready
document.dispatchEvent(new CustomEvent('tool-ready', {
    detail: { name: 'My Tool', version: '1.0.0' },
    bubbles: true,
    composed: true
}));

Best Practices

  • Always emit tool-ready after initialization
  • Emit tool-stateChanged after major state changes
  • Handle errors gracefully and emit tool-error when appropriate
  • Use bubbles: true and composed: true for events to cross Shadow DOM boundary
  • Include relevant metadata in event detail objects
  • Maintain clean state management for easy reset operations
